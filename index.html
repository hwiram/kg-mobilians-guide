3. 연동 방식

🔁 일반 결제 방식

  • 인증과 승인 절차가 결제창 내에서 자동으로 처리됨
  • 사용자가 결제창을 통해 결제를 완료하면 KG모빌리언스에서 ok_url과 noti_url로 결과 전송
  • 구현이 간단하며, 단건 결제 또는 단순 상품 판매에 적합

✅ 사용 조건: hybrid_pay: "N" 또는 미입력 (거래등록 요청 시 옵션)

  1. 거래 등록 (registration API)
  2. pay_url로 결제창 호출
  3. 사용자 결제 진행
  4. 결제 완료 시 자동 승인 처리 → ok_url/noti_url로 결과 전달
  5. CP는 SUCCESS 응답 시 서비스 제공

🟡 유의사항: 결제 실패/중복 결제 방지를 위한 noti_url 로직 방어 권장

🔀 하이브리드 결제 방식

  • 결제창에서는 인증만 수행
  • 승인은 CP 서버에서 API로 직접 요청
  • 복합결제(여러 상품), 예약결제, 승인 시점 지연 등 다양한 시나리오에 유리

✅ 사용 조건: 거래등록 요청 시 hybrid_pay: "Y" 로 설정
결제창에서 인증 후 CP가 /api/approval API 호출 필요

  1. 거래 등록 (hybrid_pay=Y)
  2. pay_url로 결제창 호출
  3. 사용자 결제 (인증만 수행됨)
  4. 인증 완료 후 ok_url/noti_url로 결과 수신
  5. CP가 pay_token으로 승인 API 호출
  6. 승인 결과 확인 후 서비스 제공

4. 테스트 및 검증

1. 🔐 결제 흐름 개요

  1. 거래 등록
    - API : POST /MUP/api/registration
    - 주요 파라미터 : sid, cash_code (CN: 카드, MC: 휴대폰), amount, trade_id, ok_url, noti_url
  2. 결제창 호출
    - 응답받은 pay_url을 팝업/iframe 으로 호출
    - 예시: window.open("https://mup.mobilians.co.kr/MUP/api/payment.mcash?tid=xxxx");
  3. 결제 진행 및 승인
    - 사용자 결제 후 KG모빌리언스는 ok_url과 noti_url로 결과 전송
    - ok_url: 프론트 화면 처리용
    - noti_url: 서버 간 통신, 필수로 SUCCESS 출력해야 함

2. 🔧 테스트 환경 구성

  • 테스트 서버 도메인: https://test.mobilians.co.kr
  • 테스트 가능한 결제수단: 휴대폰결제, 가상계좌
    👉 신용카드: 운영환경에서 실제 승인 테스트 필요

3. 🧪 주요 테스트 항목

  1. 거래등록 → 결제창 호출 → 결제 진행 전체 플로우 확인
  2. ok_url 응답 및 noti_url 서버간 통신 결과 검증 (code == '0000', hmac 무결성 확인)
  3. 결제금액, 상품명, 사용자정보 등 실제 입력값 정상 전달 여부
  4. FAIL 응답 처리 시나리오도 함께 검토 필요 (결제 실패, 사용자 취소 등)

4. 📩 결제 결과 처리 (noti_url)

  • 응답값 : SUCCESS 또는 FAIL (텍스트만 출력)
  • SUCCESS 출력 시 조건
    • code == '0000'
    • IP가 KG모빌리언스 서버인지 확인
    • hmac, amount 등 응답 항목 검증 일치
  • ※ SUCCESS 출력 누락 시 최대 20회 재요청 → 중복 서비스 주의!

5. 🔁 결제 취소 및 환불

  • 공통 API: POST /MUP/api/cancel
  • cash_code 구분 필요
  • 전체/부분취소 구분: part_cancel: Y/N
  • ⚠️ amount(금액): 휴대폰결제 부분취소 금액 셋팅 방법 필수 체크
